T6.4 Exception trace_id: FAIL

Expected: Unhandled exception (RuntimeError) error log contains trace_id
Actual: Error log has NO trace_id (empty string)

Evidence:
  status: 500 (correct)
  response has traceparent: True (correct - 00-588afa6e...-2335c0d6...-01)
  log has trace_id: False (FAIL)

Root Cause (see f3-t6-error-trace-diagnostic.txt for full analysis):
  Starlette's build_middleware_stack (applications.py:62-66) assigns the
  Exception (catch-all) handler to ServerErrorMiddleware (OUTERMOST middleware),
  which runs OUTSIDE ContextMiddleware's `with request_context(...)` block.
  Therefore, ctx.get("trace_id") returns "" in the global exception handler.

  In contrast, AppError-specific handlers are registered on ExceptionMiddleware
  (INNERMOST middleware, INSIDE ContextMiddleware), so they CAN read trace_id.
  This was confirmed by comparing call stacks:
    - AppError handler: called from _exception_handler.py:59 (route level)
    - Exception handler: called from middleware/errors.py:176 (ServerErrorMiddleware)

Impact:
  - Business errors (AppError): trace_id IN log ✓
  - Unhandled exceptions (RuntimeError, ValueError, etc.): trace_id NOT IN log ✗
  - Response traceparent: correctly set for BOTH (from request.scope) ✓

Fix Path:
  In global_exception_handler (basic_tool/errors/handler.py), extract trace_id
  from request.scope["basic_tool.traceparent"] instead of ctx.get("trace_id"):
    traceparent = request.scope.get("basic_tool.traceparent", "")
    trace_id = traceparent.split("-")[1] if len(traceparent.split("-")) >= 2 else ""
  This is the same scope-based approach already used for the response header.
